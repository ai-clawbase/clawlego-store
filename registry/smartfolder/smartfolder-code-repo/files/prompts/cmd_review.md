# /review — 评审改动或指定路径

用法：`/review`（评审当前未提交改动）或 `/review <相对路径>`（评审某文件/目录）。

## 工作流

1. 确定评审范围：
   - 无参数 → 在 `${SPACE_ROOT}` 下跑 `git diff`（含 staged：`git diff HEAD`）拿改动；不是 git 仓库则提示用户指定路径。
   - 有路径 → 读该文件/目录的源码。
2. 结合 `${SPACE_ROOT}/.llm/index/` 的依赖信息，判断改动是否影响调用方、是否漏改关联处。
3. 按这几类逐条过：
   - **正确性**：逻辑错误、边界条件、错误处理、并发/资源泄漏
   - **复用/简化**：是否重复造轮子、能否调用现成函数、是否过度设计
   - **一致性**：是否贴合周边代码的命名/惯用法/注释密度
   - **风险**：破坏性操作、外向副作用、缺测试
4. 输出评审结论：每条给 `file:line` + 问题 + 建议；按严重度排序（阻断 > 应改 > 可选）。先给 1–2 句总评。

## 产物归属

- 评审报告是**本轮对话产物**：写到 `${CONV_ROOT}/outputs/review_<时间或范围>.md`，**不**写进代码库的 `.llm/`。
- 评审是只读分析；**不要**擅自改代码。用户认可某条建议后，再走 persona 的"改源码必须先确认"流程实施。

## 规则

- 只报有把握的问题；不确定的标"待确认"并说清依据缺口
- 引用一律 `file:line`，路径相对 `${SPACE_ROOT}`
