把一个代码目录"提升"为智能代码库后，`/map` 会扫描全部源码，给每个文件生成摘要、抽出导出符号与依赖关系，索引落在 `.llm/index/` 下，可被任意对话和后续 worker 直接读取。之后你可以用自然语言检索、让 agent 解释任意模块、对改动做评审——不必每次都把整个仓库塞进上下文。

# 你能用它做什么

- "这个项目的 HTTP 路由都在哪里注册的？"
- "解释一下 syncer 这个包是怎么工作的"
- "找出所有调用 AtomicWriteFile 的地方"
- "评审一下我刚改的这几个文件有没有问题"
- "新人接手该从哪个文件读起？"

# 落盘结构（agent 维护）

```
<代码库根>/
├── cmd/ internal/ ...          ← 源码，agent 只读（改动必先确认）
└── .llm/
    ├── manifest.yaml
    ├── about.md
    ├── index/                  ← 索引产物（git-ignored，可随时重建）
    │   ├── files.jsonl         ← 每个文件一行：摘要 / 符号 / 依赖
    │   └── symbols.jsonl       ← 每个导出符号一行：定义位置 / 签名
    └── state/
        └── map_cursor.json     ← 增量水位（path → sha256）
```

# 产物归属（重要）

- **索引/符号表/水位**是 `/map` 跑出来的**可重建缓存**，落 `.llm/{index,state}/`。
- **评审报告、生成的文档、重构方案**等本轮产物，一律落**会话** `<CONV_ROOT>/outputs/`——代码库内**没有** outputs 目录。
- 确需把某份成果长期归档进代码库时，由你显式写到一个明确的源码/文档路径（如 `docs/`），而不是堆进缓存目录。
